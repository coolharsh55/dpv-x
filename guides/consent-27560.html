27560-DPV Records

Overview

A consent record, as defined earlier, is a documentation of information associated with consent. In [[ISO-27560]], a consent record contains four sections: header for specifying metadata regarding the record, processing containing information regarding the processing activities associated with that consent, parties for information on involved entities, and events for provenance of activities associated with that consent.

In [[DPV]], the concept ConsentRecord represents a consent record. Due to way DPV is utilised and expressed (e.g. RDF graph, JSON) - there is no necessity to replicate the four sections of [[ISO-27560]] as information is linked and expressed as a single graph. Therefore, the DPV implementation using ConsentRecord only requires the existence of mandatory information fields and ignores the structuring into sections.

Header Fields

The header fields represent metadata about the record and includes the following 3 fields:

Schema Version

the schema used to interpret the record and its information. For DPV, this is represented using dct:conformsTo with a reference to the schema ID or IRI. To represent the DPV consent record schema, the value of this field should be https://w3id.org/dpv/schema/27560-record. To represent the DPV GDPR consent record schema, the value of this field should be https://w3id.org/dpv/schema/27560-GDPR-record.

This field MUST be present.

schema version

Record Identifier

the unique identifier for this record. For DPV, this is represented using dct:identifier with a UUID-4 string as value. This field MUST be present in both [[27560]] and the DPV implementation.

Using schema version

Data Subject

the reference to data subject (pii principal id in [[ISO-27560]]) the consent record is about. For DPV, this is represented using dpv:hasDataSubject with an instance of DataSubject or a reference or identifier to an instance. This field MUST be present in both [[27560]] and the DPV implementation.

Processing Fields

[[ISO-27560]] contains 22 fields related to processing activities, and 5 additional fields regarding personal data involved in processing. The structuring of these fields within [[ISO-27560]] is of the form where the "PII Processing" section contains an array of "purposes" where each "purpose" is expressed with its own fields regarding legal basis, collection method, storage locations, and so on. Within the DPV implementation, this is replicated using PersonalDataHandling instances. Therefore, each of the below specified fields are used through an instance of PersonalDataHandling.

purpose

the purpose of processing personal data, expressed in DPV using hasPurpose as an instance of Purpose. This field MUST be present in both [[27560]] and the DPV implementation.

purpose type

the category of stated purpose, expressed in DPV by using concepts from Purpose taxonomy and associated with the purpose using dct:type. The purpose category provides a broad context to the purpose and aids in interoperability by providing a common framing through the taxonomy. This field MAY be present in [[ISO-27560]] implementations but MUST be present in DPV implementations.

legal basis

the legal basis (lawful basis in [[ISO-27560]] for processing of personal data. Though being a consent record, the legal basis field ensures that the processing is justified using consent, and also enables the consent record structure to be reused to other legal bases (e.g. contract) in the future. For the DPV implementation, this is expressed using hasLegalBasis with a concept from the LegalBasis taxonomy. For expressing jurisdiction specific legal basis, the appropriate extension should be used, for e.g. using consent types from [[DPV-GDPR]]. This field MAY be present in both [[ISO-27560]] and DPV implementation.

data controller(s)

the data controllers (pii controllers in [[ISO-27560]]) responsible for the specified processing activities, expressed in DPV by using hasDataController with instances of DataController. This field MUST be present in both [[ISO-27560]] and DPV implementations.

service(s)

the services this processing activities are part of or assist with, expressed in DPV using hasService and an instance of Service.

This field MAY be present in both [[ISO-27560]] and DPV implementations.

personal data

the personal data (pii information in [[ISO-27560]]) associated with processing activities, expressed in DPV using hasPersonalData with PersonalData. This field MUST be present in both [[ISO-27560]] and DPV implementations. In [[ISO-27560]], the personal data fields are additionally describe the nature of personal data as follows:

In the fields here, dct:type is used to denote types or categories, e.g. for purposes and personal data, instead of the typical RDF semantics which use rdf:type, rdfs:subClassOf, or even skos:broader/skos:narrower. This is because using any of these implies a specific interpretation of categories, which may not be compatible with the rest of the information. For example, expressing category using rdf:type would make it an instance of that category which it may not necessarily be. If a specific DPV profile is being used, then it can opt to use the appropriate semantic relation in addition to the one used here.

  • type: the type or category of personal data (pii type in [[ISO-27560]]), expressed in DPV using dct:type, and optionally using concepts from the [[DPV-PD]] taxonomy. This field MUST be present in both [[ISO-27560]] and DPV implementations.
  • identifier: an identifier representing the data values or instances e.g. within a database, expressed in DPV using dct:identifier. This field MAY be present in both [[ISO-27560]] and DPV implementations.
  • necessity: the necessity or optionality of this data (pii optional in [[ISO-27560]]) for the processing, expressed in DPV using hasNecessity and one of the provided instances of Necessity. This field MAY be present in both [[ISO-27560]] and DPV implementation, and its absence is to be interpreted as meaning the data is necessary.
  • sensitive personal data: whether the personal data is considered sensitive (sensitive pii category in [[ISO-27560]]), denoted in DPV by using dct:type with value SensitivePersonalData. This field MAY be present in both [[ISO-27560]] and DPV implementations.
  • special category personal data: whether the personal data belongs to one of special categories based on jurisdictional regulations such as GDPR (special pii category in [[ISO-27560]]). It is expressed in DPV implementations by using dct:type with value SpecialCategoryPersonalData. This field MAY be present in both [[ISO-27560]] and DPV implementations.

    Considerations

    Security and Privacy

    • record is personal data - information within a consent record and the record itself also constitutes personal data, therefore appropriate measures are required to safeguard it
    • identification for record/receipt - each consent must be attributed to the data subject it is about, however this does not mean records and receipts require additional identification procedures to get this information. Where no identity or identifier is available, a new identifier should be created for the purposes of record/receipt only e.g. visitors to the website
    • integrity of contents - mechanisms such as using cryptographic hashes to provide integrity checks for information, ensuring storage retains accuracy of information, and any changes to structure of information should be assessed to ensure interpretation of contents of records/receipts is not changed
    • verification of contents - disputes on changes in contents and discrepancies between records or receipts held by two or more entities are possible; mechanisms to cryptopgraphically 'sign' the information using verifiable means can assist in preventing such issues. The current norm within 27560 is Controllers creating records and providing receipts to the data subjects who can then use it to keep their own records. This guide does not presume this is the only possible use-case, and considers any entity
    • transmission / communication - between controllers
    • communication of receipt to data subjects

    Information Modelling

    • for controllers - storing full records/receipts is not required
    • for controllers - separate records or single records
    • for controllers - updating vs appending
    • for data subjects - creating own records
    • for data subjects - storing and using receipts

    • DNT, GPC

    Additional Schemas and Profiles

    • records - GDPR specific, notice only i.e. only mention the notice and record consent events (the notice has to be 27560 conformant)
    • receipts - A13/A14, signals only, full+explicit i.e. mention not applicable in fields instead of blanks
